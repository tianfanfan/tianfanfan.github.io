## 什么是执行上下文（Execution Context）

执行上下文是针对函数而言，很多书和网上文章都有对其的描述，但是并没有非常官方的百科描述，各种书籍对其描述也不一致。究其原因在于，执行上下文，实际上是随着语言、或年代、或完善程度在不断的变化。

那么 `Javascript` 执行上下文到底包含多少东西呢？根据部分网上 `Ecma` 规范。和 `winter` 分享：

##### 执行上下文在 ES3 中，包含三个部分。

- `scope`：作用域，也常常被叫做作用域链。
- `variable object`：变量对象，用于存储变量的对象。
- `this value`：this 值。

##### 在 ES5 中，我们改进了命名方式，把执行上下文最初的三个部分改为下面这个样子。

- `lexical environment`：词法环境，当获取变量时使用。
- `variable environment`：变量环境，当声明变量时使用。
- `this value`：this 值。

##### 在 ES2018 中，执行上下文又变成了这个样子，this 值被归入 lexical environment，但是增加了不少内容。

- `lexical environment`：词法环境，当获取变量或者 this 值时使用。
- `variable environment`：变量环境，当声明变量时使用。
- `code evaluation state`：用于恢复代码执行位置。
- `Function`：执行的任务是函数时使用，表示正在被执行的函数。
- `ScriptOrModule`：执行的任务是脚本或者模块时使用，表示正在被执行的代码。
- `Realm`：使用的基础库和内置对象实例。
- `Generator`：仅生成器上下文有这个属性，表示当前生成器。

##### 上下文和作用域的关系

`javascript` 里面作用域是`词法作用域`，依赖是语法，即函数没执行就会确定，作用域链是一个链表。

根据上面的解释，其实也就解释了，**执行上下文其实和作用域的关系是包含和被包含的关系**，但是往往人们提及他们，说的是一个概念，因为上下文里面除了作用域，**其他一些内容，在正常使用时，几乎不会被注意到**。

但其实，是有一些差异，作用域是函数声明就确定，执行上下文，是执行才创建。

总之，**上下文就是函数执行过程中可以操作的指针，内存。**

这些指针被放入到一个大对象里面，即`executionContextObj`。

它包含了函数传入的参数，自身申明的变量，以及作用域链，等等等。

## 闭包是什么？（Closure）

知道了函数执行上下文，作用域，闭包就很好理解了，

闭包用自己的话概况，就是：**函数返回值，可以访问到其部分作用域(scope) 的现象**。

技术名词官方解释是：一个函数 `A`，返回值是另一个函数 `B` 时候，函数 `B` 内部可以读取函数 `A` 的作用域，这个可访问被关闭的函数作用域的函数 `B`，叫闭包。

也就是说，官方解释，闭包是这个被返回的函数。

那么，实行起来就很简单了:

```js
function closerBuilder() {
  var slot = 1
  return function closer() {
    return slot
  }
}
var close = closerBuilder()
```

上例子中，函数 `closerBuilder`，返回值是另一个函数 `closer` ，函数 `closerBuilder` 执行结束后，函数 `closer` 内部可以读取函数 `closerBuilder` 的作用域（变量 `slot`），函数 `closer` 叫闭包。

然而，实际上，`javascript` 作为一门高级语言，任何其他类型的返回值都可能访问 `closerBuilder` 的作用域。举个简单的例子，返回一个对象，对象内部的所有属性，或者方法，都可以访问 `closerBuilder` 的内部作用域。
所以说，**其实，写 JS ，时时刻刻都在写闭包**。

## 调用栈（Context Stack）

函数执行，会把创建出的执行上下文，放入一个栈（数组）里，执行结束，会剔除。

函数执行过程中又调用了另外一个函数，会再放入一个执行上下文到栈里。

1. ##### 官方名词解释

   在**函数执行时候又调用了另外一个函数**，会把代码的执行流进入内部函数，将创建一个新的执行上下文，并把它压入现有栈的顶部。
   `javascript` 引擎永远会执行当前栈中顶部的执行上下文 一旦函数在当前执行上下文执行完毕，它会被从栈的顶部弹出，然后将控制权移交给当前栈的下一个上下文当中。

   要注意以下几点：

   1. 有函数执行，就有栈。
   2. 栈不是无限大，栈会爆。即，函数内部调用另一个函数，层次太深，就会爆栈。
   3. 一个函数执行结束 return 或其他方式结束，栈就会释放一个。

2. ##### 栈和执行上下文的关系

   栈是保存着执行上下文的数组，是引擎内部管理函数执行上下文的一种方案。

   **栈会保存正在执行的函数的执行上下文**。

3. ##### 栈和作用域链的关系

   `javascript` 里面作用域链依赖是语法，即词法作用域，即函数没执行就会确定，是一个链表。再深也不会爆掉。

   而 `javascript` 调用栈是指函数在执行时候才出现的，执行上下文保存的地方，是一个数组。太长会爆掉。

   **栈和执行上下文有关系，执行上下文和作用域链有关系。**

   栈和作用域链，二者之间，没关系直接关系。

## 递归

1. ##### 普通递归

   顾名思义，就是函数自己内部调用自己。
   一般说递归，都是说同步递归。如下代码

   ```js
   function task() {
     // console.log('task')
     task()
   }
   task()
   ```

   `task` 会一直创建执行上下文，一直加入到栈里面，

   示例代码中，这个递归，没有结束条件，会爆栈。ERROR: Maximum call stack size exceeded。

   虽然每次的 `task` 的执行上下文的内容是相同的，但是引擎并不知道的一件事是，执行过程会不会出现内容不同的上下文。所以干脆直接放进去。这种做法也是正确的。

2. ##### 异步递归

   异步递归是指，递归过程，掺杂着异步。

   ```js
   function task() {
     console.log("task")
     setTimeout(task, 0)
   }
   task()
   ```

   虽然也是无限执行下去，但是这个执行，栈不会爆。也不会卡顿。

   仔细观察，这里其实是两个函数在一直调用，同步函数 `task` 和，把函数加入异步队列的函数 `setTimeout。`

   这里要注意，**setTimeout 函数本身是同步的**。

   > `setTimeout` 接收两个参数，

   > 第一个参数是函数，作用是把第一个参数，加入到 `js` 引擎的任务队列里面。

   > 之后函数结束，并返回一个 `id` 的标记。

   这段代码的整个执行过程：

   1. 从第一次开始，`task` 执行，调用栈深度 `+1`。
   2. 执行 `setTimeout`，调用栈深度 `+1`，`js` 引擎的任务队列 `+1`。
   3. `setTimeout` 调用结束 调用栈深度 `-1`。
   4. `task` 结束，调用栈深度 `-1`。
   5. 引擎事件轮循。
   6. 引擎是否有时间执行队列里面的任务了。
   7. 有，引擎队列 `-1`
   8. 开始执行 `task`，回到第一步。

   <!-- 这里实际上有两个栈，一个是调用栈，一个是任务队列。

   - 调用栈把下一个任务加入任务队列，然后清除调用栈。

   - 任务队列在有时间执行的时候，把任务拿出来，并清除任务队列。

   这样一来，调用栈，等于是一直使用调用栈末尾一个位置。甚至都不是一个调用栈里了。 -->

   异步递归，不算真正的递归。因为递归是指函数内部执行自己。

   而异步递归，并不是函数内部执行自己，是把自己加入任务队列。函数就结束了。

   **所以，不爆栈其实很简单，那就是让函数 return。**

   而不是让函数一直执行另一个函数。

## 尾调用和尾递归

1. ##### 尾调用

   名词解释

   函数的有效语句最后一个语句，执行了另一个函数。

   ```js
   function task2() {}

   function task() {
     return task2()
   }
   task()
   ```

2. ##### 尾递归

   名词解释

   函数的有效语句最后一个语句，调用了自己。

   ```js
   function task() {
     return task()
   }
   task()
   ```

3. ##### 尾递归优化

   名词解释

   尾递归情况下，不再爆栈，让栈可以释放。

   因为尾递归的上一次的执行上下文，确实是没用的，不会再回来了。

   现阶段一些浏览器做到了。但是很多是不会的。[各大 js 引擎特征支持性][ct6]

   其实也可以理解，因为尾递归没必要优化。
